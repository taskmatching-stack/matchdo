# 封測版真實廠商功能 — 可行性評估

**評估前提**：先不寫程式，僅評估是否能在**不影響現有穩定系統**下建立封測廠商功能。  
**結論**：✅ **可行**。建議**初期採用種子廠商模式**，不引入認領／試用／到期等複雜流程；若日後要擴充為「白手套入駐與認領」，再依下文第二～六節以加法方式落地。

---

## 〇、採用「種子廠商模式」（初期封測）

**什麼是種子廠商模式**  
由平台**主動邀請或挑選**少量真實廠商，在後台（或既有流程）**直接建檔**：建立 `manufacturers`、綁定 `user_id`（廠商登入帳號）、上傳 3～5 組作品／對照圖／AI 圖。這批廠商就是「種子廠商」，用於封測階段的真實曝光與流程驗證。

**與白手套認領的差異**

| 項目 | 種子廠商模式（建議先用） | 白手套認領（可選 Phase 2） |
|------|---------------------------|-----------------------------|
| 建檔方式 | 平台／營運在後台直接建立廠商並綁定帳號，或廠商自己註冊後建廠商資料 | 平台代建「未認領」店鋪，產生專屬連結，廠商點連結認領後綁定 |
| 狀態與到期 | **不需要** account_status / expires_at / claim_token；沿用現有 `is_active`、`verified` 即可 | 需要 Private → Trial → Premium / Suspended 與 60 天倒數 |
| 權限分級 | 可先**全部給滿**（等同現有行為），或只做簡單開關（例如種子廠商一律可編輯、可上傳白模） | 需權限矩陣（封測體驗版唯讀／翻譯限額／付費解鎖） |
| 對現有系統影響 | **幾乎為零**：只是「多幾筆由你們建立的廠商資料」，走既有 API 與前台 | 需新增欄位、認領頁、權限攔截、限額引擎 |

**種子廠商模式具體做法（不寫程式，只列作法）**

1. **人選**：挑選 N 家真實廠商（例如 5～15 家），取得對方同意與基本資料（名稱、介紹、聯絡方式、作品圖等）。
2. **建檔**：  
   - 若廠商已有本站帳號：在後台或透過既有「建立廠商資料」流程，以該帳號建立一筆 `manufacturers`，並代為上傳 3～5 組作品（或提供連結／圖檔由你們上傳）。  
   - 若廠商尚未註冊：先為其建立 auth 帳號（或請對方註冊），再建一筆廠商並綁定 `user_id`，同上傳作品。
3. **曝光**：維持現有邏輯即可。`is_active = true` 的廠商會出現在首頁、搜尋、廠商列表；必要時可設 `verified = true` 標示為認證廠商。
4. **不新增**：不需認領連結、不需 Trial/Premium 狀態、不需 60 天倒數、不需依方案攔截 API 或翻譯限額。權限矩陣可留作**日後正式收費時**再實作。

**小結**  
初期的「封測版真實廠商」用**種子廠商模式**即可達成：在現有架構下多幾筆真實廠商資料與作品，**零額外開發、零破壞既有穩定**。下文第二～六節保留原「白手套入駐與認領」的完整評估，供日後若要改為認領制或分級付費時參考。

## 一、現有系統簡要現狀（與本需求相關）

| 項目 | 現況 |
|------|------|
| 廠商表 | `public.manufacturers`（見 `docs/custom-products-schema.sql`）：有 `id`, `name`, `description`, `categories`, `contact_json`, `location`, `is_active`, `verified`, `user_id`（綁定登入帳號）等。**目前沒有**：帳號狀態枚舉、到期日、認領用 token。 |
| 曝光邏輯 | 首頁／搜尋／廠商列表等處多以 `is_active = true` 篩選廠商；未區分「草稿／試用／付費／過期」。 |
| 權限 | 廠商端 API 多依「是否登入 + 是否為該廠商之 user_id」判斷，**沒有**依「方案等級」或「狀態」做功能開關。 |
| 點數／翻譯 | 現有方案與點數機制存在；AI 翻譯若有使用，目前應無「依方案限額」的邏輯。 |

因此，新機制需要**新增欄位與新流程**，並在**讀取與寫入處**加上「狀態／方案」判斷，但**不必改動既有欄位語意**，只要對「既有資料」給予預設值即可保持相容。

## 二、四大模組可行性與對現有系統影響

### 1. 廠商狀態管理 (Vendor Account Status)

| 項目 | 評估 |
|------|------|
| **可行性** | ✅ 高。在 `manufacturers` 上**新增**欄位即可，例如：`account_status`（枚舉：private / trial / premium / suspended）、`expires_at`（TIMESTAMPTZ, 可 NULL）、必要時再加 `plan_tier`（如 trial / lv3 / lv4）以對應權限表。 |
| **不影響穩定的做法** | （1）新欄位皆設 **default** 或遷移時寫入預設值：**既有廠商一律設為 `premium`（或等同現有行為）**，且 `expires_at = NULL`，這樣現有曝光與後台行為不變。（2）所有「會曝光廠商」的查詢（首頁、搜尋、廠商列表）改為：`is_active = true` **且** `account_status IN ('trial', 'premium')`（或等同邏輯）；Private / Suspended 不列入。 |
| **風險** | 低。唯一要注意：遷移腳本必須對**既存資料**明確寫入 `account_status = 'premium'`（或你們決定的「視同付費」值），避免舊資料因 NULL 被新邏輯當成「未認領」而隱藏。 |

---

### 2. 專屬認領連結機制 (Claim URL & Token)

| 項目 | 評估 |
|------|------|
| **可行性** | ✅ 高。新增一組「認領用」資料即可：例如在 `manufacturers` 加 `claim_token`（UNIQUE, 可 NULL）、`claimed_at`、`claimed_by_user_id`；或獨立表 `manufacturer_claim_tokens`（manufacturer_id, token, expires_at）。白手套建好廠商後寫入一筆 token，系統產生連結：`/claim?token=xxx`。 |
| **流程** | 訪客開連結 → 後端依 token 只查得到 `account_status = 'private'` 的廠商（且 token 吻合）→ 顯示預覽頁（店鋪＋AI 圖）→ 點「認領此店鋪」→ 登入／註冊（綁定 email 或手機）→ 後端將該廠商改為 `user_id = 當前使用者`、`account_status = 'trial'`、`expires_at = now() + 60 天`、清除或作廢 token。 |
| **不影響穩定的做法** | （1）**僅「白手套代建」的廠商**才會有 claim_token、才走認領流程；既存廠商沒有 token，不受影響。（2）認領頁與 API 皆為**新增**（例如 GET/POST `/claim` 或 `/api/claim-vendor`），不修改既有廠商 API 的語意。 |
| **風險** | 中低。須確保：token 不可猜（夠長、隨機）、一廠商一 token、認領後 token 失效；且認領時要防重複綁定（例如該 manufacturer 已有 `user_id` 則不允許再認領）。 |

---

### 3. 權限攔截器 (Feature Gating / Middleware)

| 項目 | 評估 |
|------|------|
| **可行性** | ✅ 高。前端與後端都做一層「方案／狀態」判斷即可。 |
| **後端** | 在「會寫入或執行付費功能」的 API（例如：新增/編輯作品、上傳對照圖、AI 去背、排行榜查詢等）開頭取得該廠商的 `account_status` / `plan_tier`，若為 trial 且該功能在權限表為「需付費」，則回傳 **403 或 402** 與明確錯誤碼（例如 `UPGRADE_REQUIRED`），前端據此彈出升級彈窗。 |
| **前端** | 登入後若可取得廠商狀態（例如從現有「我的廠商」API 或新增欄位帶出），則在「新增作品」「編輯」「AI 去背」「排行榜」等按鈕或頁面加上判斷：trial 時改為顯示升級彈窗（不直接發送會失敗的請求）。 |
| **不影響穩定的做法** | （1）**僅在「狀態為 trial（或未付費）」時**才做攔截；premium / 既有廠商（已設為 premium）不受限。（2）攔截邏輯是**加一層 if**，不刪改既有成功路徑。 |
| **風險** | 中。必須**清單化**所有「付費專屬」API 與按鈕，避免漏攔；否則 trial 用戶可繞過前端直接呼叫 API。建議：後端集中一個「權限檢查 helper」，依 `manufacturer_id` + 功能代碼判斷，凡寫入/敏感讀取都經此關卡。 |

---

### 4. 點數與翻譯限額引擎 (Quota Limiter)

| 項目 | 評估 |
|------|------|
| **可行性** | ✅ 高。點數：Trial 可於認領成功時由後端**一次性**寫入一筆點數（例如 500 點），沿用現有點數表或流水表。翻譯：在呼叫「AI 翻譯」的 API 前，查詢該使用者／廠商當期「已使用翻譯句數」，若 ≥ 10 句且為 trial，則回傳額度用盡、前端顯示「需升級才能繼續跨國對話」。 |
| **不影響穩定的做法** | （1）**僅對 trial（或未付費方案）** 套用翻譯上限；premium / 既有廠商不限制（或依現有點數扣抵即可）。（2）若目前沒有「翻譯句數」統計，需新增一筆表或欄位記錄使用量；不更動既有點數扣抵邏輯，只是多一個維度。 |
| **風險** | 低。需定義清楚「一句」的單位（例如：一則送出訊息內含 N 句、或一次 API 呼叫算一句），並在產品與工程上一致。 |

## 三、權限矩陣（定價定錨表）與實作對應

您提供的矩陣（封測體驗版 / LV3 / LV4）可直接對應到「狀態 + 方案等級」：

- **account_status = 'trial'** → 封測體驗版（60 天、唯讀作品集、體驗點數、翻譯 10 句上限、不可上傳白模、不可看排行榜等）。
- **account_status = 'premium' + plan_tier = 'lv3'** → 品牌曝光包（自由編輯、對照圖、白模 5 款、翻譯依點數等）。
- **account_status = 'premium' + plan_tier = 'lv4'** → 全球商機雷達（白模 30 款、排行榜解鎖等）。

實作時可做一張「功能代碼 ↔ 所需最低方案」對照表（例如：`upload_comparison` → lv3；`leaderboard_access` → lv4；`vendor_assets_count` → 5 或 30 依方案），後端與前端的權限檢查都依這張表，方便日後擴充與定價頁展示。

## 四、不影響現有穩定系統的條件總覽

1. **既有廠商一律視為「已付費」**  
   遷移時將現有 `manufacturers` 的 `account_status` 設為 `premium`（且 `expires_at = NULL`），或等同「維持目前所有權限」的狀態，這樣現有廠商在首頁、搜尋、後台行為完全不變。

2. **新邏輯以「加法」為主**  
   只新增欄位、新表、新路由（認領頁、認領 API）、新判斷分支；不刪除既有欄位、不改變既有 API 的「成功時」回傳格式與語意。

3. **曝光與搜尋條件明確**  
   任何「會把廠商列給訪客看」的查詢（首頁、搜尋、廠商列表、圖庫等），都加上：只顯示 `account_status IN ('trial','premium')` 且未過期（若 `expires_at` 有值則 `expires_at > now()`）。Private 僅能透過帶 token 的認領連結看到；Suspended 一律不曝光、後台只顯示升級結帳頁。

4. **認領僅限「白手套建立的廠商」**  
   只有帶 `claim_token` 且 `account_status = 'private'` 的廠商才出現在認領流程；既存廠商沒有 token，不會被誤認領。

5. **功能開關只對「未付費」狀態生效**  
   Trial（與未來若有的其他未付費狀態）才套用：作品唯讀、翻譯上限、白模上傳禁止、排行榜隱藏、付費功能 API 回傳 402/403。Premium 與既有遷移後的廠商不受影響。

## 五、風險與建議

| 風險 | 建議 |
|------|------|
| 漏攔截某個 API，Trial 可繞過前端操作 | 後端建立「廠商方案權限檢查」helper，清單列出所有受保護的 API，一律經此檢查；新 API 上線時強制經過同一關卡。 |
| 既有資料遷移漏設 status，導致被隱藏 | 遷移腳本明確：`UPDATE manufacturers SET account_status = 'premium' WHERE account_status IS NULL`（或等效），並在部署前於測試環境驗證首頁／搜尋仍看得到既有廠商。 |
| 認領 token 外洩或重複使用 | Token 足夠長（如 32 位元隨機）、認領後即清空或標記已使用；同一 manufacturer 僅允許認領一次（已有 user_id 則拒絕）。 |
| 翻譯「句」定義不清，日後爭議 | 在規格與 API 文件寫明：一句 = 一次翻譯 API 呼叫／或一則訊息內 N 句，並在 UI 顯示「已使用 x/10 句」。 |

---

## 六、總結

- **初期封測**：建議採用**種子廠商模式**（見第〇節）。不新增認領／狀態／到期／權限矩陣，只由平台建檔並綁定真實廠商，零額外開發、不影響現有穩定。
- **若日後改為白手套認領或分級付費**：第一～五節的四大模組與權限矩陣仍適用。在「既有廠商預設為 premium、新規則只套在 trial／private／suspended」的前提下，皆可以**加法**落地；建議順序：**（1）狀態與到期日欄位 + 遷移** → **（2）認領連結與流程** → **（3）後端權限攔截** → **（4）前端升級彈窗與翻譯／點數限額**。
- 權限矩陣（封測體驗版 / LV3 / LV4）可留作日後定價頁與方案對照表，待正式收費時再實作功能開關。
